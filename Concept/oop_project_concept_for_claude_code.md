# OOP курс C# — Концепція проєкту та Git стратегія
## Документ для Claude Code

---

## Контекст

Викладач розробляє **еталонний навчальний проєкт** на C# для курсу OOP.
Еталон пишеться на домені **Бібліотека** — потім транспонується в загальні інструкції для студентів.

Студент працює **автономно**:
- має свій GitHub репозиторій
- робить коміт після кожного завдання
- сам зливає гілки в main коли готовий
- викладач аналізує гілки асинхронно — не блокує прогрес

## Правила Git

### Гілки
```
main                        ← завжди робочий стан системи
sandbox/intro               ← тема 1
sandbox/arrays              ← тема 2
feature/[назва-лаби]        ← нова функціональність
refactor/[назва]            ← покращення без нової функціональності
hotfix/[назва]              ← виправлення знайденого багу
```

### Правила роботи
```
1. Кожне виконане завдання → обов'язковий коміт
2. Коміт-повідомлення: "Lab03 Task2: Add Author class with validation"
3. Нова лаба стартує від актуального main
4. Студент зливає сам коли готовий — не чекає викладача
5. Гілки не видаляються — це доказ роботи
6. Викладач аналізує гілки асинхронно
```

### Коміт після кожного завдання — обов'язково
Це дозволяє викладачу бачити:
- коли студент працював
- як розвивалось рішення
- де виникли труднощі (багато правок одного файлу)
- чи є різкий стрибок якості (ознака зовнішньої допомоги)

## Правило злиття в main

**Зливається одразу** — якщо після merge система вміє щось нове що видно в консолі:
- нове меню або команда
- нова поведінка існуючої команди
- підключений новий модуль

**Зливається разом з наступною лабою** — якщо зміна внутрішня і не впливає на поведінку консолі (рефакторинг, покращення структури).

**Студент не чекає дозволу** — зливає сам. Викладач аналізує факт після.

---

## Структура кожної лаби: 4 рівні складності

Кожна лаба — це один модуль або точкове розширення існуючого.
Всередині лаби — 4 завдання які **послідовно нарощують** функціонал.

| Рівень | Характер | Вимога | Оцінка |
|---|---|---|---|
| Завдання 1 | Базова реалізація | Обов'язкове | Залік |
| Завдання 2 | Розширення, взаємодія компонентів | Обов'язкове | Достатньо |
| Завдання 3 | Нетривіальне рішення, наближення до наступної теми | Бажане | Добре |
| Завдання 4 | Відкрита проблема — студент сам знаходить рішення | Бонус | Відмінно |

### Важливо про завдання 3 і 4

Завдання 3 і 4 **не є простими**. Вони вимагають:
- самостійного пошуку підходу
- розуміння всього попереднього коду
- прийняття архітектурного рішення

Формулюються як **проблема**, не як технічне завдання:

❌ Погано: "Реалізуйте патерн Strategy для політики позики"

✅ Добре: "Бібліотека хоче підтримувати різні правила позики для різних категорій читачів. Стандартний читач — максимум 3 книги, преміум — 10, дитячий — тільки дитячі книги. Як зробити так щоб додавання нової категорії не змінювало існуючий код?"

Студент сам приходить до патерну. Це цінніше ніж знати назву патерну.

Завдання 4 кожної лаби **навмисно веде до концепції наступної теми** — студент відкриває її сам до пояснення викладача.

## ConsoleApp — живе весь курс

`Program.cs` це точка входу яка **ніколи не переписується** — тільки розширюється.

```csharp
// Лаба 03 — просте меню
while (true)
{
    Console.WriteLine("1. Show all books");
    Console.WriteLine("2. Add book");
    Console.WriteLine("0. Exit");
    // ...
}

// Лаба 12 — підключились нові модулі
// але структура меню та сама — просто більше пунктів
// і виклики до нових сервісів
```

Студент бачить як система росте через єдину точку входу.

## Як аналізувати роботу студента

Викладач заходить в репозиторій студента і бачить:

```
Що перевіряти в гілці:
  ✓ кількість комітів (1 коміт = списав, 5-7 = працював)
  ✓ дати комітів (все в останній день = підозра)
  ✓ diff від main (тільки те що стосується лаби чи є зайве)
  ✓ чи є завдання 3 і 4 (показує рівень студента)
  ✓ стиль коду відповідає попереднім лабам

Червоні прапорці:
  ⚠ один великий коміт перед дедлайном
  ⚠ код значно складніший ніж попередні лаби
  ⚠ використані концепції яких ще не було в курсі
  ⚠ коментарі або стиль відрізняється від попередніх лаб
```

## Як Claude Code має працювати з цим проєктом

### Порядок роботи
```
1. Читаємо цей документ повністю
2. Читаємо існуючі лаби в папці labs/ (якщо є)
3. Дивимось поточний стан src/ щоб розуміти де ми знаходимось
4. Пишемо наступну лабу або інструкцію
```

### Для кожної лаби Claude Code створює

**Еталонний код** — повна реалізація на домені Бібліотека:
```
src/Modules/[ModuleName]/    ← якщо новий модуль
або точкові зміни в існуючих файлах
```

**Інструкцію** у форматі MD:
```
labs/lab-NN-[назва]/
 └── instructions.md
```

### Структура instructions.md для кожної лаби

```markdown
# Лаба NN — Назва теми

## Мета
Одне речення що студент навчиться робити.

## Контекст
Що вже є в системі. Яку проблему вирішує ця лаба.

## Гілка
git checkout main
git pull
git checkout -b feature/[назва]

## Завдання 1 — [назва] (обов'язкове)
Конкретний опис. Що створити. Що має працювати.
Обов'язковий коміт: "Lab-NN Task1: [опис]"

## Завдання 2 — [назва] (обов'язкове)
...
Обов'язковий коміт: "Lab-NN Task2: [опис]"

## Завдання 3 — [назва] (бажане)
Формулюється як проблема. Не підказує рішення.
Обов'язковий коміт: "Lab-NN Task3: [опис]"

## Завдання 4 — [назва] (бонус)
Відкрита проблема. Студент сам знаходить підхід.
Обов'язковий коміт: "Lab-NN Task4: [опис]"

## Злиття
git checkout main
git merge feature/[назва]
git push

## Що має з'явитись в консолі після злиття
Конкретний опис нової поведінки програми.

## Питання для самоперевірки
- Питання 1
- Питання 2
- Питання 3
```

### Важлива деталь інструкцій

Інструкції пишуться в **загальному вигляді** — без конкретних назв домену Бібліотека. Використовуються абстрактні назви:

```
Сутність A    замість    Book
Сутність B    замість    Author  
Користувач    замість    Member
Операція      замість    Loan
Каталог       замість    BookCatalog
```

Студент підставляє свій домен. Еталонний код на Бібліотеці є як приклад — але в інструкції його немає.
